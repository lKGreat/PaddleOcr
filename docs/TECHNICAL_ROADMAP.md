# 长期技术路线图

- 最近更新：2026-02-07
- 维护人：项目维护团队

## 1. 文档目标
用于统一未来 12-18 个月的技术演进方向，确保架构决策、功能建设与业务目标一致，作为迭代规划与技术评审基线。

## 2. 总体技术目标
- 工程化：形成稳定的 .NET OCR 工具链（训练、导出、推理、服务联调）。
- 产品化：核心命令可直接支撑业务场景，输出结果可追溯、可集成。
- 平台化：模块可插拔（`det/rec/cls/e2e`），便于按场景扩展。

## 3. 分阶段路线
## Phase A（0-3 个月）：可用性夯实
- 完善 `pocr` 核心命令稳定性与错误分级。
- 统一配置加载与覆盖规则（`-c` / `-o`）。
- 建立最小回归测试集与标准样例数据。

## Phase B（3-6 个月）：能力增强
- 增强 ONNX 推理链路性能与可观测性（耗时、关键日志、失败原因）。
- 打通训练-导出-推理的一致配置体验。
- 输出标准化结果结构，便于业务系统消费。

## Phase C（6-12 个月）：平台化扩展
- 建立模块化扩展规范（新模型、新后处理、新任务类型）。
- 增加服务端对接能力与批处理任务治理能力。
- 完善资产版本管理（配置、字典、样例）与兼容策略。

## 4. 技术基线与度量
- 质量：关键链路必须有自动化测试，新增功能需补齐回归用例。
- 性能：建立端到端耗时基线，迭代中持续监控波动。
- 可维护性：公共接口稳定、模块边界清晰、异常语义统一。

## 5. 风险与应对
- 模型与配置耦合高：通过配置视图与校验降低误配风险。
- 资产变更频繁：通过 `scripts/sync-assets.ps1` 和版本记录保证一致性。
- 需求变化快：采用小步提交与阶段验收，确保快速回滚与定位。

## 6. 维护机制
- 更新频率：每个迭代至少更新一次。
- 更新要求：每次更新需说明“变更原因、影响范围、验收方式”。
- 责任分工：功能 owner 更新对应阶段条目，评审人负责一致性检查。
